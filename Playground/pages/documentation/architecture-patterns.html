LLM Guide - Architecture Patterns

Architecture Patterns

How to structure your NUI application using the Router.

The NUI Router (nui.createRouter and nui.enableContentLoading) is a flexible, hash-based routing system. A common misconception is that the router requires every page to be a separate HTML file with its own isolated <script type="nui/page"> block.

Important: While the NUI Playground uses the "Fragment-Based" pattern heavily for demonstrations, it is not mandatory. The router is fundamentally just a state coordinator that translates URL hashes into DOM visibility.

This guide outlines the three primary ways to architect an application using the NUI Router.

Pattern 1: Centralized Application Logic (The "App" Pattern)

In this pattern, your application logic lives in a central JavaScript bundle. The router is simply used to toggle the visibility of different pre-rendered or dynamically generated DOM elements based on the URL state.

How it works

  • All HTML is either in the main index.html or generated dynamically by a central script.
  • You register custom route handlers using nui.registerFeature or nui.registerType.
  • No HTML fragments are fetched over the network.

Example

// main.js
import { nui } from './nui.js';

// Register a feature that generates its own UI
nui.registerFeature('dashboard', (element, params) => {
    element.innerHTML = `
        <header><h1>Dashboard</h1></header>
        <div class="stats">Loading stats...</div>
    `;
    
    // Centralized logic handles the data fetching
    appState.fetchStats().then(data => {
        element.querySelector('.stats').textContent = data;
    });
});

// Register another feature that just shows an existing DOM element
nui.registerFeature('settings', (element, params) => {
    // Clone a template from the main document
    const template = document.getElementById('settings-template');
    element.appendChild(template.content.cloneNode(true));
    
    // Bind centralized event listeners
    appSettings.bindForm(element.querySelector('form'));
});

// Start the router
const router = nui.createRouter(document.querySelector('nui-main'), {
    default: 'feature=dashboard'
});
router.start();

When to use

  • Complex, highly interactive applications (like dashboards or editors).
  • Apps with shared state (e.g., a shopping cart) that needs to be accessed across different views.
  • When you want to bundle all your JavaScript together for performance and immediate execution.

Pattern 2: Fragment-Based Pages (The "Playground" Pattern)

This is the pattern used by the NUI Playground. The router fetches HTML files from the server and injects them into the DOM. Each fragment can contain its own scoped JavaScript.

How it works

  • You use nui.enableContentLoading() which automatically sets up a page route type.
  • Navigating to #page=about fetches /pages/about.html.
  • If the HTML contains <script type="nui/page">, it is extracted and executed once with a scoped init(element, params) function.
⚠️ Critical for LLMs: The <script type="nui/page"> pattern is non-standard. Browsers ignore this script type. NUI manually extracts and executes it after upgrading custom elements. See Router Deep Dive below for the full execution model.

Example

<!-- pages/about.html -->
<h1>About Us</h1>
<button class="track-btn">Click Me</button>

<script type="nui/page">
function init(element, params) {
    // Logic is scoped entirely to this fragment
    const btn = element.querySelector('.track-btn');
    btn.addEventListener('click', () => console.log('Clicked on About page'));
    
    // Optional lifecycle hooks
    element.show = () => console.log('About page became visible');
}
</script>

When to use

  • Documentation sites, blogs, or content-heavy sites.
  • Playgrounds and component libraries (where isolation is a feature).
  • When you want to split code automatically without a complex bundler setup.

Router Deep Dive: Page Script Execution Model

This section explains exactly how the NUI Router loads pages and executes scripts. Understanding this is critical for writing correct page code.

The Problem: Script Timing

Standard <script> tags execute immediately when HTML is parsed. But NUI pages contain custom elements (<nui-button>, <nui-tabs>, etc.) that need to be upgraded first. If scripts run too early, querySelector calls fail, event listeners attach to unupgraded elements, and everything breaks.

The Solution: Deferred Script Execution

NUI uses a custom script type that browsers ignore, then manually extracts and executes it after upgrading components.

Execution Flow (Step by Step)

1. User navigates to: #page=components/button
2. Router fetches: pages/components/button.html
3. Router creates wrapper: <nui-page class="content-page ...">
4. Router injects HTML into wrapper
5. Router calls: customElements.upgrade(wrapper)
   → All <nui-*> elements inside are now upgraded
6. Router finds: <script type="nui/page">
7. Router removes the script tag from DOM
8. Router executes the script via new Function()
9. Script's init(element, params, nui) is called automatically

The Script Tag is a Trigger, Not a Container

The <script type="nui/page"> tag signals "execute page logic now." The actual code can be loaded three ways:

Option 1: Inline (Most Common)
<script type="nui/page">
function init(element, params, nui) {
    element.querySelector('nui-button').addEventListener('nui-click', handler);
}
</script>
Option 2: Imported Module (For Complex Pages)

Import the module in main.js and use an empty trigger tag:

// main.js
import { initButtonPage } from './pages/components/button.js';
window.nuiPageInit = window.nuiPageInit || {};
window.nuiPageInit['components/button'] = initButtonPage;
 
<!-- pages/components/button.html -->
<!-- Empty trigger tag - router calls window.nuiPageInit['components/button'] -->
<script type="nui/page" data-init="components/button"></script>
Option 3: Pre-loaded Function

Define the function in a standard script tag, then call it from the trigger:

<!-- Standard script - browser parses but doesn't auto-execute -->
<script type="text/javascript">
function initButtonPage(element, params, nui) { /* ... */ }
</script>

<!-- Trigger tag calls the pre-defined function -->
<script type="nui/page">
function init(element, params, nui) {
    initButtonPage(element, params, nui);
}
</script>

Important: The router looks for <script type="nui/page"> and executes its contents. If the tag is empty or contains only a function call, that's perfectly valid. The tag's purpose is to signal "run page initialization now."

Page Caching Behavior

Pages are cached by the router. The wrapper element is created once, fetched once, and shown/hidden on navigation:

First visit to #page=components/button:
  1. Fetch HTML
  2. Create wrapper element
  3. Execute init(element, params, nui)
  4. Show page

Second visit to #page=components/button:
  1. Find cached wrapper element
  2. Call element.show(params) ← init() is NOT called again
  3. Show page

When leaving the page:
  1. Call element.hide()
  2. Hide page (display: none)

Common Mistakes (DO NOT DO THESE)

❌ Wrong: Using document.querySelector
function init(element, params, nui) {
    const button = document.querySelector('nui-button'); // WRONG!
    // May find buttons from other pages or miss the target
}
✅ Correct: Using element.querySelector
function init(element, params, nui) {
    const button = element.querySelector('nui-button'); // CORRECT!
    // Scoped to this page's wrapper
}
❌ Wrong: Attaching global listeners without cleanup
function init(element, params, nui) {
    document.addEventListener('resize', handleResize); // WRONG!
    // Listener persists after page is hidden
}
✅ Correct: Cleanup in hide()
function init(element, params, nui) {
    function handleResize() { /* ... */ }
    
    document.addEventListener('resize', handleResize);
    
    element.hide = () => {
        document.removeEventListener('resize', handleResize); // CORRECT!
    };
}
❌ Wrong: Assuming script re-runs on navigation
function init(element, params, nui) {
    let counter = 0; // WRONG! This won't reset on navigation
    // Because init() only runs once
}
✅ Correct: Reset state in show()
function init(element, params, nui) {
    let counter = 0;
    
    element.show = (params) => {
        counter = 0; // CORRECT! Reset when page becomes visible
    };
}

Lifecycle Hooks Summary

Hook When Called Use For
init(element, params, nui) First page load only Setup, attach listeners, cache elements
element.show(params) Every time page becomes visible Start timers, reset state, fetch data
element.hide() Every time page is hidden Stop timers, cleanup, save state

Why This Pattern Exists

  • Custom elements must be upgraded first - Standard <script> runs too early
  • Pages are cached for performance - init() runs once, show()/hide() handle visibility
  • Scoped DOM access - element parameter prevents cross-page interference
  • Zero dependencies - No framework, no build step, pure browser APIs

Pattern 3: The Hybrid Approach

The NUI Router allows you to mix both patterns seamlessly. You can have a centralized app that handles complex features, while using the fragment loader for static content like help pages, documentation, or terms of service.

How it works

  • You use nui.enableContentLoading() to handle #page=... routes automatically.
  • You use nui.registerFeature() to handle #feature=... routes with centralized logic.

Example

// main.js
import { nui } from './nui.js';

// 1. Centralized logic for the complex app feature
nui.registerFeature('editor', (element, params) => {
    element.innerHTML = '<div id="canvas"></div>';
    const editor = new ComplexEditor(element.querySelector('#canvas'));
    
    element.show = (params) => editor.loadDocument(params.docId);
});

// 2. Enable content loading for static pages (Help, About, etc.)
nui.enableContentLoading({
    container: 'nui-main',
    basePath: '/pages',
    defaultPage: 'home'
});

// Now the app responds to both:
// #feature=editor&docId=123 -> Runs centralized logic
// #page=help/shortcuts      -> Fetches /pages/help/shortcuts.html

When to use

  • Almost any real-world application.
  • When you want to keep your main JavaScript bundle small by offloading static content to HTML fragments, while keeping core application logic centralized.

Understanding the Trade-offs

Each pattern exists because different applications have different needs. The key is understanding what you're optimizing for:

When Centralized Logic Makes Sense

Applications with complex state management, real-time collaboration, or heavy interactivity benefit from keeping logic in one place. When multiple views need access to the same data (a shopping cart, a document editor's undo history), fragment-based isolation becomes a hindrance rather than a help. Centralized logic also eliminates network requests for view transitions, which matters for apps where responsiveness is critical.

When Fragment Loading Shines

Content-heavy sites — documentation, blogs, marketing pages — often have discrete pages that don't share state. Here, fragment loading provides automatic code splitting without build tool complexity. Each page can be edited independently, cached intelligently by the browser, and even generated by a CMS without touching the application JavaScript.

The Router's Role

Regardless of which pattern you choose, the router's job remains the same: it creates a wrapper div, caches it, and calls element.show() / element.hide() as the URL changes. How that div gets populated — via fetch or JavaScript generation — is an implementation detail that the router leaves entirely up to you.

State Lives in the URL

One invariant across all patterns: the URL hash (#type=id) determines what is shown, and the search params (?key=value) provide the data for that view. This means bookmarkable state, back-button support, and shareable links come free with any architecture choice.
